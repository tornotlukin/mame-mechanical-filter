# mame-mechanical-filter

Two small Python CLI tools for curating a MAME romset and exposing it to RetroArch with proper game titles. Pure stdlib, no third-party dependencies.

## Why this exists

A full MAME romset is enormous and full of machines you'll never play — mechanical fruit machines, gambling cabinets, ROMs flagged unrunnable, NAOMI/Dreamcast units that need their own core, touchscreen-only games, and so on. RetroArch makes it worse: since MAME 0.280, its bundled CRC32 database (`MAME.rdb`) doesn't recognise newer ROMs, so scanning the romset shows raw filenames like `sf2.zip` instead of `Street Fighter II`.

This repo solves both problems with two pipeline stages:

1. **`distill/`** — filter a MAME romset, copying only what you actually want to a clean folder.
2. **`lpl-builder/`** — build a RetroArch playlist for that folder with proper game titles in the `label` field, bypassing the broken scan entirely.

---

## Requirements

- **Python 3.10+** — standard library only, nothing to `pip install`.
- **A local MAME install whose version matches your romset.** The tools are version-agnostic: they read game data straight from *your* `mame.exe`, so a 0.288 romset needs a 0.288 MAME, a 0.275 romset needs 0.275 MAME, and so on.
- **Your MAME romset** as a folder of `.zip` files. **Merged** or **non-merged** sets are easiest; **split** sets work too (see the romset-type section).

---

## Quick start

Everything is keyed to **your MAME version** — point the commands at your own `mame.exe` and romset folder. On Windows, run from **PowerShell or cmd**, not Git Bash (see the path gotcha near the end).

```
# --- Do these three once per MAME version (re-do them after a MAME upgrade) ---

# 1. Dump your MAME's full game database to XML.
H:\mame\mame.exe -listxml > lpl-builder\mamegames.xml

# 2. Derive every filter/preservation list (bios, devices, mechanical, naomi, chd, clones)
#    from that XML. This is the script that "makes the lists from MAME".
python distill/build_lists.py --xml lpl-builder/mamegames.xml

# 3. Download the matching catver.ini (genre data, drives the catver-based filters).
python distill/distill.py -dl

# --- Do these whenever you want to (re)build a curated set ---

# 4. Filter your full romset into ./clean (source is never modified).
python distill/distill.py --source X:\path\to\your\MAME_ROMS --dest ./clean

# 5. Build the RetroArch playlist, with the device path your ROMs will live at.
python lpl-builder/build.py --source ./clean --device-prefix "/storage/emulated/0/RetroArch/roms/mame"
```

Then copy `./clean/` and `lpl-builder/MAME.lpl` onto your device — see **[Installing on RetroArch](#installing-on-retroarch)**.

> **Version match matters.** `mamegames.xml`, the derived lists, and `catver.ini` must all come from the **same MAME version as your romset**. Mismatched versions produce wrong game names and miss newly-added machines. Whenever you upgrade MAME, re-run steps 1–3.

## Important: romset type (merged / split / non-merged)

**This matters more than anything else here.** Whether a game still runs after you delete other zips depends entirely on how your romset is packed. A MAME game zip is rarely self-contained — it leans on other zips:

- **Parent ROMs** — a clone shares most of its data with its parent game.
- **BIOS ROMs** — e.g. every Neo Geo game needs `neogeo.zip`.
- **Device ROMs** — shared subcircuits with their own ROM data, e.g. `qsound.zip` (the QSound DSP used by every CPS-2 game), Namco custom MCUs, CPU decryption keys.

The three pack types handle those dependencies differently:

| Type | Clones | BIOS / device ROMs | What distill does |
|---|---|---|---|
| **Split** (MAME's native format) | Clone has its own small zip containing only its *differences*; needs the parent zip present. | Separate zips. | Each kept clone's **parent zip is pulled in** automatically (even if the parent's category is excluded), and parents are protected from `--prune`. |
| **Merged** | Clone ROMs are packed *inside* the parent zip; clones have no standalone zip. | Separate zips. | Clones travel with their parent for free; no parent handling needed. |
| **Non-merged** | Every game (parent *and* clone) has its own zip containing **everything** it needs, BIOS and device ROMs included. | Embedded in each game zip. | Every zip is standalone; no parent handling needed. Largest on disk, simplest to curate. |

**How distill handles all three.** distill filters by zip filename and adapts to the pack type:

- It **preserves BIOS and device ROMs by default** for every set type (see below) — without that, pruning silently deletes `qsound.zip` and friends and a huge swath of games stop booting.
- On a **split** set it resolves the clone→parent dependency: each kept clone's parent zip is copied alongside it, and prune won't delete a parent a kept clone still needs. The clone→parent map comes from `lists/clones.json` (generated by `build_lists.py`).
- The pack type is **auto-detected** from the source folder (it samples whether clone zips exist as standalone files). Override with `--set-type {merged,split,non-merged}` if detection guesses wrong. Auto resolves an ambiguous "clones present" set to **split**, which is the safe superset — pulling parents on a genuinely non-merged set is harmless redundancy.

If games still fail to launch after distilling, the cause is almost always a missing BIOS or device zip — confirm your source set type and that preservation is on (it is, unless you passed `--exclude-bios` / `--exclude-devices`).

## Tool 1: `distill`

Walks a folder of MAME ROM zips and copies a filtered subset to a destination, dropping anything that matches the exclusion categories.

### What it filters by default

| Category | Source | Why excluded |
|---|---|---|
| `mechanical` | `lists/mechanical_only.xml` (`ismechanical=yes`) + catver `Electromechanical` | Physical reel/coin machines that don't play meaningfully in software emulation. |
| `fruit` | catver `Casino / Slot Machine` and `Electromechanical / Reels` | Slot machines. |
| `gambling` | catver `Casino/*` | Casino card/table games. |
| `touch` | `lists/touch_only.xml` | Touchscreen-only games, unusable without a touchscreen. |
| `naomi` | `lists/naomi_only.xml` (NAOMI-family `sourcefile`) | NAOMI / Atomiswave / Dreamcast ROMs — these belong to a different core. |
| `computer` | catver `Computer / *` and `Computer Graphic Workstation` | Home computers (Apple II, C64, Atari ST, etc.) — not arcade games. |
| `console` | catver `Game Console / *` and `Game Console/Computer` | Home consoles (NES, SNES, Genesis, etc.) — belong in their own emulator. |
| `chd` | `lists/chd_only.xml` (own non-nodump `<disk>`) + filesystem scan | CHD-requiring games. Skipped unless `-chd` is passed (then the `.zip` and its sibling `.chd` folder are treated as a unit). |

> **No `nonrunnable` category.** An earlier version filtered a `nonrunnable_only.xml`, but that list turned out to be **100% devices** (`isdevice=yes`) — so it was really deleting device ROMs like `qsound` that games depend on, which is why games stopped booting after distilling. Devices are now *preserved*, not excluded, and the category is gone.

### BIOS and device preservation

Two classes of dependency ROM are **always copied** even when they also match another exclusion category, because games reference them as external zips (see the romset-type section above):

- **BIOS** — machines flagged `isbios="yes"` (e.g. `neogeo.zip`, `decocass.zip`, `naomi.zip`, `pgm.zip`). Listed in `lists/bios_only.xml`.
- **Devices** — machines flagged `isdevice="yes"` that carry ROM data (e.g. `qsound.zip` for CPS-2 games, Namco custom MCUs, CPU decryption subboards). Listed in `lists/devices_only.xml`.

Without this, an exclusion filter would delete these dependency zips — `qsound`, for instance, is a `runnable="no"` device yet is required by every CPS-2 title — and the games that need them stop booting. Both lists, along with `mechanical_only.xml`, `naomi_only.xml`, `chd_only.xml`, and `clones.json`, are derived from MAME's own XML in one pass:

```
python distill/build_lists.py --xml lpl-builder/mamegames.xml
```

Regenerate after every MAME version bump — this is the single command that refreshes every list the distiller derives from MAME. (`touch_only.xml` is the one list not derived here, since MAME has no clean touchscreen flag; `catver.ini` is downloaded separately with `distill.py -dl`.) Pass `--exclude-bios` and/or `--exclude-devices` to turn off preservation (only safe on a fully non-merged set).

### Usage

```
python distill/distill.py --source X:\Arcade\roms\MAME287_ROMS --dest ./clean
```

### Flags

| Flag | Effect |
|---|---|
| `--source PATH` | Source ROM folder. Default: current directory. |
| `--dest PATH` | Destination folder. Default: `<source>/_distilled/`. |
| `-chd`, `--chd` | Include CHD-requiring games. Each game's `.zip` plus its matching `.chd` subfolder is copied together. |
| `-dl`, `--download` | Download the latest `catver.ini` from progettoSNAPS and exit. Run this once before your first distill so genre matching works. |
| `--keep-mechanical` | Don't exclude mechanical games. |
| `--keep-fruit` | Don't exclude fruit / slot games. |
| `--keep-gambling` | Don't exclude `Casino/*` games. |
| `--keep-touch` | Don't exclude touchscreen games. |
| `--keep-naomi` | Don't exclude NAOMI / Dreamcast ROMs. |
| `--keep-computer` | Don't exclude home computer ROMs. |
| `--keep-console` | Don't exclude home console ROMs. |
| `--exclude-bios` | Drop the BIOS-preservation override; BIOSes can then be filtered by any active category. |
| `--exclude-devices` | Drop the device-preservation override; ROM-bearing devices can then be filtered. Only safe on a fully non-merged set. |
| `--set-type {auto,merged,split,non-merged}` | Romset pack type. `auto` (default) detects merged vs split from the source. `split` pulls each kept clone's parent zip along with it and protects parents from prune; `merged`/`non-merged` need no parent handling. |
| `--prune` | Destructive mode: walk `--source` and DELETE files that match active exclusions, instead of copying. Use after the filter rules change to clean a previously-distilled folder. Defaults to dry-run; pair with `--yes` to actually delete. `--dest` is rejected in this mode. |
| `--yes` | Confirm destructive prune deletions. Required alongside `--prune`. |
| `-v`, `--verbose` | Debug logging. |

### Behaviour notes

- **Source is never modified.** Files are copied with `shutil` (or `robocopy` on Windows when the network share misbehaves). The original folder is read-only as far as the script is concerned.
- **Resumable.** If a destination file already exists with the same size, it's skipped — so re-running after a crash or interruption picks up where it left off without re-reading hundreds of GB.
- **SMB-tolerant.** Reads use a chunked loop instead of `shutil.copy2`'s `readinto` path (which throws `OSError 22` on some shares). When even that fails, the copy falls back to Windows `robocopy`. If both fail for a specific file it's logged in the final summary and the run continues instead of aborting.

### Pruning an existing distilled folder

When filter rules change (e.g. you add `computer`/`console` exclusions after an earlier distill), `/clean` won't auto-update — distill only copies, it doesn't delete. Use `--prune` to clean it up in place:

```
python distill/distill.py --source ./clean --prune          # dry-run, shows counts
python distill/distill.py --source ./clean --prune --yes    # actually delete
```

Source folders at the real ROM location (e.g. `X:\Arcade\roms\...`) are never touched by anything else — `--prune` only deletes from whatever you pass as `--source`.

### Layout

```
distill/
  distill.py       # CLI entry
  config.py        # constants
  loaders.py       # parse lists/*.xml and *.txt
  catver.py        # parse catver.ini
  exclusions.py    # build per-category exclusion sets, classify a ROM
  copier.py        # enumerate ROMs, copy zips + CHD subfolders
  downloader.py    # fetch latest catver.ini from progettoSNAPS
  build_lists.py   # derive bios/devices/mechanical/naomi/chd lists + clones.json from mamegames.xml
  romset.py        # detect pack type (merged/split/non-merged), resolve clone->parent deps
```

## Tool 2: `lpl-builder`

Builds a RetroArch `MAME.lpl` playlist directly from a folder of ROMs plus MAME's `-listxml` output. Each playlist item gets the proper game title pre-populated in its `label` field, so RetroArch shows real names without doing any CRC matching.

### Why a hand-built playlist?

RetroArch's normal flow is: scan a ROM, hash it, look up the hash in `MAME.rdb` (a bundled snapshot of an older MAME `-listxml`), and use the database's name as the label. Post-MAME 0.280 the bundled database lags behind, so any newer ROM scans as the raw filename. Since `label` is just a regular JSON field, the simplest fix is to skip the scan entirely and write the playlist with correct labels from the start.

The schema, including the top-level envelope and item field order, was verified against `playlist_write_file()` in `libretro/RetroArch` master — see `docs/learned_retroarch-playlist-format.md` for the full crib sheet.

### Prerequisite: generate the MAME XML

```
H:\mame\mame.exe -listxml > lpl-builder\mamegames.xml
```

The XML is large (~300 MB) and gitignored. The script streams it with `xml.etree.iterparse` so memory stays flat.

### Usage

```
python lpl-builder/build.py --source ./clean --device-prefix "/storage/emulated/0/RetroArch/roms/mame"
```

### Flags

| Flag | Effect |
|---|---|
| `--source PATH` | Folder of ROM `.zip` files to enumerate. Required. |
| `--xml PATH` | Path to MAME `-listxml` output. Default: `lpl-builder/mamegames.xml`. |
| `--device-prefix STR` | Path prefix written into each item's `path` field, joined with forward slashes. Use the Android (or other target) ROM path. When unset, the actual PC path of each ROM is used — handy for testing on desktop RetroArch. |
| `--output PATH`, `-o` | Where to write the `.lpl`. Default: `lpl-builder/MAME.lpl`. |
| `-v`, `--verbose` | Debug logging. |

### Output

A single `MAME.lpl` JSON file. Top-level shape:

```json
{
  "version": "1.5",
  "default_core_path": "",
  "default_core_name": "",
  "label_display_mode": 0,
  "right_thumbnail_mode": 0,
  "left_thumbnail_mode": 0,
  "thumbnail_match_mode": 0,
  "sort_mode": 0,
  "items": [
    {
      "path": "/storage/emulated/0/RetroArch/roms/mame/10yard.zip",
      "label": "10-Yard Fight (World, set 1)",
      "core_path": "DETECT",
      "core_name": "DETECT",
      "crc32": "00000000|crc",
      "db_name": "MAME.lpl"
    }
  ]
}
```

`core_path` / `core_name` are `"DETECT"` so RetroArch picks the active MAME core at runtime — portable across Play Store, sideload, and aarch64 builds. `db_name` is `"MAME.lpl"` so existing thumbnail packs at `thumbnails/MAME/Named_Boxarts/...` match without extra work.

The top-level `default_core_path` and `default_core_name` are emitted as empty strings. If you'd rather hard-bind every entry to a specific core instead of relying on `DETECT`, fill them in by hand after the tool runs:

- **`default_core_path`** — the full filesystem path to the core's `.so` on the target device. There is no portable answer: Android handhelds (Retroid, Anbernic, phones) usually hide the OS root filesystem from the file picker, so you may have to dig in RetroArch's own UI to discover the actual install path. In RetroArch on the device: Settings → Directory → Cores will show you the directory; the `.so` filename you want is whatever shows up under Online Updater → Core Downloader → MAME. Common shapes look like `/data/data/com.retroarch.aarch64/cores/mame_libretro_android.so`, but the exact path depends on which RetroArch build (Play Store / aarch64 / sideload) is installed and may not be reachable through the system file manager.
- **`default_core_name`** — the human-readable core label. For recent MAME builds this is typically `"Arcade (MAME/Arcade)"`. Older or alternate cores use different strings (e.g. `"Arcade (MAME 2003-Plus)"`); check Settings → Core in RetroArch to read off the exact name.

### Layout

```
lpl-builder/
  build.py         # CLI entry
  config.py        # constants
  xml_reader.py    # stream-parse mamegames.xml
  writer.py        # build playlist dict, JSON write
  mamegames.xml    # generated by `mame -listxml`, gitignored
```

### Gotcha: Git Bash on Windows mangles the device prefix

If you pass `--device-prefix "/storage/emulated/0/..."` from Git Bash on Windows, MSYS path translation rewrites the leading `/` to `C:/Program Files/Git/storage/...`. Run from PowerShell or `cmd.exe`, or set `MSYS_NO_PATHCONV=1` if you must use Git Bash.

## Installing on RetroArch

Two outputs go to **two different places** on the target device (PC, Android handheld like a Retroid/Anbernic, or phone):

### 1. The ROMs → your MAME content folder

Copy the contents of `./clean/` (all the `.zip` files, plus the `.chd` subfolders if you used `-chd`) to wherever RetroArch will look for MAME ROMs on the device. **This path must exactly match the `--device-prefix` you gave `lpl-builder`** — the playlist stores an absolute path per game, so if the ROMs live somewhere else every entry is a dead link.

Common Android locations:

| Build / image | Typical MAME ROM path |
|---|---|
| Sideloaded RetroArch | `/storage/emulated/0/RetroArch/roms/mame/` |
| Many Retroid / Anbernic images | `/storage/emulated/0/Roms/MAME/` |
| SD card | `/storage/XXXX-XXXX/Roms/MAME/` (the `XXXX-XXXX` is the card's volume id) |

### 2. The playlist → RetroArch's `playlists/` folder

Copy `lpl-builder/MAME.lpl` into RetroArch's playlist directory. It depends on the build:

| RetroArch build | Playlists path |
|---|---|
| Play Store (`com.retroarch`) | `/storage/emulated/0/Android/data/com.retroarch/files/playlists/` |
| Play Store 64-bit (`com.retroarch.aarch64`) | `/storage/emulated/0/Android/data/com.retroarch.aarch64/files/playlists/` |
| Sideloaded / older builds | `/storage/emulated/0/RetroArch/playlists/` |
| Desktop (Windows) | `%APPDATA%\RetroArch\playlists\` or the portable install's `playlists\` |

**Find the exact path in-app:** RetroArch → Settings → Directory → Playlists. Whatever's shown there is where `MAME.lpl` belongs.

### 3. Load it

Open RetroArch and the **MAME** playlist appears in the sidebar with proper game names. Picking a game loads it through the MAME core (`DETECT` lets RetroArch choose the installed one). If you want box art, drop a thumbnail pack into `thumbnails/MAME/Named_Boxarts/<label>.png` (filenames match the playlist `label`) — this works automatically because the playlist's `db_name` is `MAME.lpl`.

> Building for a device from a PC? Set `--device-prefix` to the **device** path (e.g. `/storage/emulated/0/...`). The PC path where `./clean` happens to sit is irrelevant — only the device path is written into the playlist. Omit `--device-prefix` only when you're testing on the same PC where the ROMs live.

## Re-running and upgrading MAME

The two tools are independent — re-run either stage on its own when its inputs change:

- **Upgraded MAME to a new version?** Re-run Quick Start steps 1–3 (dump XML → derive lists → download catver) so everything matches the new romset, then re-distill.
- **Changed the filter rules** (e.g. added/removed a `--keep-*`)? Re-distill into a fresh folder, or prune the existing one in place: `python distill/distill.py --source ./clean --prune --yes` (dry-run first without `--yes`).
- **Just want a new playlist** (e.g. different device path)? Re-run only `lpl-builder/build.py`.
